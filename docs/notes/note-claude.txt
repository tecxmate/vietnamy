⏺ New study_import schema is better — but only after light normalization                                            
                                                                  
  Why it wins on coherence and scale:                                                                               
                                                                                                                    
  1. Single source of truth for Vietnamese text. word_text/sentence_text is byte-identical across all 5 language    
  bases. Adding a 6th language (Thai, Indonesian) means one new translation column, not a new curriculum tree. The  
  current app duplicates vi strings inside every locale block — drift is inevitable.                                
  2. Track segmentation matches the product. A1+A2 shared core → B1 forks into Manager/Tourist/Heritage is a real   
  pedagogical reality. The current app pretends 3 modes are peers with overlapping topics, which is why professional
   and heritage sit empty — the model can't express "shared prerequisite."
  3. Exercise generation is decoupled from authoring. content_types is a hint, not authored exercises. One row of   
  vocab fuels listen-choose, MCQ, match-pairs, and fill-blank without re-authoring. The current lessons.json        
  hard-codes dialogue lines and accepted translations — every new exercise type means a content rewrite.
  4. Flat schema scales linearly. 13,582 rows in 27 files is reviewable in Excel by non-engineers (which is who's   
  authoring it). The current 10-sheet relational layout requires FK discipline that breaks the moment a non-engineer
   edits it.                                   
  5. Placement test fits the same shape. Profile questions, diagnostics, and track selectors all use the same 16    
  columns with match_text/match_target carrying signal weights. Extending the system doesn't mean a new schema.     
                                               
  Where it's weaker — and what to add on import:                                                                    
                                                                  
  - No stable lesson ID. Identity is (unit_title, lesson_type, lesson_title) — fragile if titles get renamed. Fix:  
  assign deterministic hashed IDs at import time and persist them.
  - No explicit ordering, XP, or unlock rules. Current app needs node_id, quiz_id, xp_reward, unlock_rule. Fix:     
  derive from row order at import; store in a sidecar progression table, not in the curriculum file. Keeps authoring
   concerns separate from gamification concerns.
  - No grammar tag references. Current vn_grammar_bank_v2.json and GT*** tags are gone. Fix: keep grammar tags as a 
  separate dimension; auto-link by matching category or grammar pattern keywords during import.                     
  - Audio/image columns empty. Not a schema flaw — TTS pipeline fills them later, keyed by word_text/sentence_text.
  Cleaner than the current /audio/xin_chao.mp3 strings hard-coded in JSON.                                          
                                                                  
  Recommended target architecture:                                                                                  
                                                                  
  content/                       (authored — Excel/CSV, single source of truth)
    curricula/*.xlsx             ← study_import schema as-is                                                        
    translations/{lang}.json     ← derived: vi_text → translation map                                               
                                                                                                                    
  generated/                     (built — never hand-edited)                                                        
    lessons.db / lessons.json    ← lessons grouped, IDs hashed, ordering assigned                                   
    progression.json             ← node_id, quiz_id, xp, unlock_rule                                                
    grammar_links.json           ← auto-tagged grammar references                                                   
                                                                                                                    
  The study_import schema becomes the input layer; the current app's lesson/node/progression structures become a    
  derived output layer rebuilt by the converter. Authors edit Excel, engineers edit the converter and the runtime — 
  the two never collide.                                                                                            
                                                                  
  Verdict: migrate to study_import as the authoring format, keep a richer normalized form as the runtime artifact.  
  Don't try to use study_import directly at runtime (you'd lose IDs and progression) and don't try to keep the
  current normalized JSON as the authoring format (it doesn't scale to 5 languages × 4 tracks).

  How to play through new lessons                                                                                    
                                                                                                                     
  1. Start dev server: npm run dev:all                                                                               
  2. Open: http://localhost:5173/admin/curriculum-preview          
  3. Pick a mode + base language, click any lesson on the left                                                       
  4. Click the orange "Play" button in the lesson detail header                                                      
  5. Play through generated exercises — exit anytime with the X (top-right) or ESC                                   
                                                                                                                     
  Exercise types generated automatically per lesson:                                                                 
  - MCQ — Vietnamese word, 4 translation choices (one per word with translation)                                     
  - Listen — TTS speaks the Vietnamese, you pick the meaning (auto-plays on load, replay button)                     
  - Fill-blank — Vietnamese sentence with one word blanked, pick from 4 options (with translation hint)
  - Match-pairs — 5 pairs to match (uses lesson's match_text/match_target if present, else builds from vocab)        
                                                                                                                     
  The player builds exercises on-the-fly from the lesson's words / sentences / matches arrays. Distractors come from 
  other items in the same lesson, so "noise" stays thematically relevant.                                            
                                                                                                                     
  Important guardrails:                                                                                              
  - No progress is saved — no XP, no hearts lost, no SRS state changes, no localStorage writes. Pure preview sandbox.
  - Lessons with too few translatable items show a "Not enough content" screen instead of crashing — common for      
  partial JA/KO lessons or pure-Phonetics lessons that only have IPA-style entries.                            
  - TTS uses your existing /api/tts endpoint (Google Translate proxy via the Express server), so make sure dev:all is
   running both processes.                                                                                           
  - The "Play" button is disabled (greyed out) on lessons where it can't build any exercise.
